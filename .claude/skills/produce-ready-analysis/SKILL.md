# Produce Ready Analysis

## Purpose

Use this skill when executing an analytical request from `#bih-requests` and producing a finished analytical-assessment text for `#bih-readydocs`.

The goal is not to summarize all available events.

The goal is to answer one central analytical question through a coherent argument based on a limited number of strongly relevant events.

## Core workflow

1. Read the exact REQUEST ID from `#bih-requests`.
2. Read the full request.
3. Identify:
   - topic;
   - central analytical question;
   - factual period;
   - forecast horizon;
   - requested length;
   - language;
   - specific constraints.
4. Search `#bih-events-log`.
5. Select only the strongest relevant events.
6. Build one central analytical thesis.
7. Write one coherent analytical-assessment text.
8. Publish the full final text directly to `#bih-readydocs`.
9. Verify that the publication is visible.

## Evidence selection

Prefer analytical depth over informational breadth.

Unless the request explicitly requires otherwise, use approximately 4–7 strongly relevant EVENT records.

Do not include an event merely because:
- it is politically important;
- it concerns Republika Srpska;
- it falls within the requested period.

Include an event only when it:
- supports the central thesis;
- weakens the central thesis;
- reveals a meaningful constraint;
- reveals a countervailing factor;
- materially affects the forecast.

Do not invent EVENT IDs.

## Analytical writing standard

The final text must be:

- coherent;
- concise;
- analytical;
- evaluative;
- causally connected;
- professionally written;
- readable as one finished product.

The final text must not become:

- a chronological summary;
- a list of events;
- a collection of mini-analyses;
- a broad political overview;
- a methodological demonstration;
- a sequence of disconnected topic sections.

## Required analytical logic

The text should normally follow this logic:

central thesis
→ most important observed development
→ political or institutional meaning
→ causal mechanism
→ effect on the main actor or process
→ constraint or countervailing force
→ likely development

Each paragraph should emerge naturally from the previous one.

Do not start a new paragraph merely because a new actor or institution appears.

Use transitions that preserve one continuous line of argument.

## Facts, analysis and forecast

Distinguish facts, analytical judgments and forecasts through wording and epistemic calibration.

Do not use repetitive labels such as:

- `Факт:`
- `Аналитичен извод:`
- `Прогноза:`

unless the request explicitly requires them.

Instead, integrate them naturally into the prose.

Never present an analytical interpretation as an established fact.

## Calibration of language

Avoid unjustified absolute formulations such as:

- proves;
- demonstrates conclusively;
- eliminates the last possibility;
- cannot;
- completely paralyzed;
- inevitable;
- certain.

Use calibrated formulations when evidence is incomplete, such as:

- suggests;
- indicates;
- is consistent with;
- increases the probability;
- narrows the available options;
- weakens the capacity;
- appears more likely;
- with moderate confidence.

Strong categorical wording is allowed only when directly supported by the evidence.

## Causal discipline

Do not merely place events next to each other.

Explain why they are connected.

Test at least one reasonable alternative explanation when the evidence permits it.

Prefer:

observed development
→ mechanism
→ consequence

over:

event A happened
→ event B happened
→ event C happened

Do not infer coordination solely from temporal proximity.

Do not infer causation solely from correlation.

## Forecasting standard

The forecast must emerge from mechanisms identified in the analysis.

It should normally include:

- most likely development;
- principal risk to that scenario;
- factor that could alter the trajectory;
- confidence level.

Do not make generic predictions.

For each major forecast judgment, ensure that the preceding analysis provides a logical basis.

## Missing information

When important information is unavailable:

- identify the gap briefly;
- explain how it affects confidence;
- do not fill the gap with speculation.

## Style

Unless the request explicitly says otherwise:

- write in Bulgarian;
- use continuous prose;
- use meaningful paragraphs;
- avoid bullets in the main analysis;
- avoid excessive headings;
- avoid inflated academic language;
- avoid journalistic rhetoric;
- avoid repetition;
- avoid unnecessary names and institutional details;
- avoid excessive background.

For short analytical products, normally target approximately 500–600 words when the request specifies that range.

## EVENT IDs and sources

Do not place EVENT IDs inside the main analytical text unless explicitly requested.

At the end add:

`Използвани източници / EVENT IDs:`

Then list only the EVENT IDs actually used.

## Slack publication

Publish the full final text directly to:

`#bih-readydocs`

Do not:
- create DOCX;
- use Google Drive;
- create PDF;
- publish only a summary;
- publish only a link.

If Slack message limits require splitting, use sequential parts and preserve the REQUEST ID in every part.

After publishing, verify through the Slack connector that the complete text is visible.

## Final quality check

Before publication confirm:

- The central question is answered directly.
- One main thesis is visible.
- The text is not an event summary.
- Paragraphs are causally and logically connected.
- Analytical interpretations are calibrated.
- No unsupported categorical claims are present.
- The forecast follows from the analysis.
- Only relevant EVENT IDs are listed.
- The requested length is respected.
